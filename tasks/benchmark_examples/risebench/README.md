# RISEBench Croissant Tasks Example

End-to-end example for **RISEBench** — Zhao et al. 2025, *"Envisioning Beyond the Pixels: Benchmarking Reasoning-Informed Visual Editing"* (NeurIPS 2025 Track on Datasets and Benchmarks). RISEBench is a 360-sample, human-annotated benchmark for reasoning-informed image editing across four reasoning categories — **Temporal**, **Causal**, **Spatial**, and **Logical** — evaluated by an LMM-as-a-Judge pipeline (GPT-4.1).

- **Dataset**: https://github.com/PhoenixZ810/RISEBench
- **Paper**: NeurIPS 2025 (no arXiv URL at time of writing)

This example exercises **both** of the runbooks at `tasks/SKILL_pdf2ct.md` and `tasks/SKILL_ct2code.md`:

- `tasks/SKILL_pdf2ct.md`: paper PDF → Croissant Tasks `TaskProblem` (+ paper-reported `TaskSolution`s, summary, validation report). Outputs in [`pdf2ct/`](pdf2ct/).
- `tasks/SKILL_ct2code.md`: `TaskProblem` + a baseline spec → implementation code. Outputs in [`ct2code/`](ct2code/).

The ct2code stage in this example is a **single-sample proof-of-concept**, not a full evaluation. The point is to demonstrate that the pdf2ct → ct2code pipeline produces a runnable, end-to-end implementation; statistically meaningful numbers are out of scope.

## Layout

```
risebench/
├── README.md                                                ← this file
├── pdf2ct/
│   ├── risebench_problem.jsonld                             ← TaskProblem (4 subtasks, OutputSpec, EvaluationSpec)
│   ├── risebench_solution_gpt-image-1_paper.jsonld          ← TaskSolution with paper-reported metrics (Table 1)
│   ├── risebench_solution_gemini-2-flash-exp_paper.jsonld   ← TaskSolution with paper-reported metrics (Table 1)
│   ├── risebench_solution_gemini-2-flash-pre_paper.jsonld   ← TaskSolution with paper-reported metrics (Table 1)
│   ├── summary.md                                           ← pdf2ct extraction summary + SHACL findings
│   └── validation_report.json                               ← machine-readable validation log
└── ct2code/
    └── gemini-2-flash-exp/
        ├── run_eval.py                                      ← Python pipeline (inference + GPT-4.1 judge)
        └── outputs/
            ├── gemini-2.5-flash-image_scores.csv            ← PoC sample's per-dimension scores
            ├── gemini-2.5-flash-image.pkl                   ← cached judge response for the PoC sample
            └── images/logical_reasoning/logical_reasoning_1.png   ← model-generated image
```

The `hf_data/` directory the script populates locally (~193 MB of HuggingFace cache + dataset metadata + images) is `.gitignore`d — it's a derived cache. The `--use-github-output` flag avoids needing it altogether for the PoC sample (everything is fetched directly from the RISEBench GitHub repo at run time).

## PoC scope

The ct2code-generated `run_eval.py` was exercised end-to-end on **one sample** to verify the pipeline runs. Two flags were added to the original ct2code output (a `--sample` selector and a `--model` override) so the smoke test could deviate from the paper's intended setup in two ways:

- **Endpoint substitution**: the paper's `gemini-2.0-flash-exp` model is no longer available on Google AI Studio. The PoC used `--model gemini-2.5-flash-image` instead.
- **One-sample subset**: only `logical_reasoning_1` was run, via `--sample logical_reasoning_1 --skip-inference --use-github-output`. With `--use-github-output`, the input image, reference image, and model output were fetched directly from the RISEBench GitHub repo, avoiding the full HuggingFace dataset clone.

The single sample failed (GPT-4.1 judge scored ApprConsistency=1, Reasoning=1, complete=0). That outcome is not informative on its own — n=1 is not an evaluation. The point of this example is **structural**: pdf2ct produced a valid `TaskProblem` and three paper-reported `TaskSolution`s, ct2code produced a runnable script, and the script executes through inference + scoring + serialization on the sample selected. Anyone wanting to run the actual evaluation can drop the `--sample` flag (and bring sufficient API quota / a working `gemini-2.0-flash-exp` endpoint).

## Reproduction (PoC sample)

```bash
cd tasks/benchmark_examples/risebench/ct2code/gemini-2-flash-exp
export GOOGLE_API_KEY=<your-key>
export OPENAI_API_KEY=<your-key>
python run_eval.py --sample logical_reasoning_1 --skip-inference --use-github-output
```

For a full 360-sample run, drop the `--sample` and `--skip-inference` flags. The script downloads the dataset into `./hf_data/` on first invocation and caches per-sample model outputs in `./outputs/`.

Required Python packages: `google-genai`, `openai`, `pillow`, `pandas`, `numpy`, `tqdm`.

## SHACL findings

The `risebench_problem.jsonld` and the three `_paper.jsonld` solution files were tested against `tasks/croissant-tasks-shapes.ttl` as currently shipped on the `mmlu_example` branch. Three distinct shape bugs in the upstream shapes file block validation:

| # | Shape | pyshacl symptom | Status vs PR #1025 |
|---|---|---|---|
| 1 | `TaskProblemShape` | `'exists but is not a well-formed SHACL PropertyShape'` — `sh:property` blank node has no top-level `sh:path` (its body is `sh:or`) | **Known** — = Luis's Bug A |
| 2 | `TaskShape` (base) | `ClassConstraintComponent: croissant:evaluation must point to an EvaluationTask` — but `TaskProblemShape` permits `EvaluationSpec`; both shapes apply additively under RDFS inference because `TaskProblem ⊑ Task` | **New** — also surfaced by the `core/` example in this PR |
| 3 | `EvaluationTaskShape` | `'QualifiedValueShapeConstraintComponent must have at least one sh:qualifiedValueShape predicate'` — `sh:qualifiedMinCount 1` used without companion `sh:qualifiedValueShape` | **Known** — = Luis's Bug B |

This is the second example in this PR (alongside `core/`) to surface bug #2 — the `TaskShape` ↔ `TaskProblemShape` evaluation conflict — strengthening the case for fixing it upstream. RISEBench solution shapes are simpler than CoRe's, so the deeply-nested `sh:or` issue (CoRe's bug #4) does not fire here.

**Spec discipline**: per PR #1025's convention, this PR deliberately **does not modify** `tasks/croissant-tasks*.ttl` or `tasks/validator.py`. Bugs are documented as upstream issues with suggested fixes in [`pdf2ct/summary.md`](pdf2ct/summary.md) and [`pdf2ct/validation_report.json`](pdf2ct/validation_report.json) for Leo to address in PR #1017.

The .jsonld files in this directory validate against a **locally-patched** shapes file applying all three fixes (semantically equivalent to the upstream intent — see `pdf2ct/validation_report.json` `upstream_issues[].suggested_fix`).
